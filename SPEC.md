# イロトイロ（Irotoiro）— 正式仕様書 v1.0（確定）

> 承認状況：①ウィンドウ列挙・②同一セル競合の選択制・③技術構成（C++コア＋WASM＋Three.js）すべて承認済み。本書は確定版。

2人用・完全情報・確率要素あり（タイル抽選のみ）の対戦型ボードゲーム。
本書はゲームエンジン／AI／GUI実装の基準となる唯一の正式仕様。実装（codex）とAI設計はすべて本書に従う。

---

## 0. 用語

| 用語 | 定義 |
|---|---|
| ボード | 5×5＝25セルのグリッド。座標は `(row, col)`、`row,col ∈ 0..4`。 |
| おはじき | 単色の駒。色は **赤(R) / 青(B) / 黄(Y)**。総数24（各色8）。 |
| タイル | 2色混合の駒。色は **紫(P)=R+B / 緑(G)=B+Y / 橙(O)=R+Y**。総数24（各色8）。 |
| 手持ち（ストック） | 各プレイヤーがまだ盤に置いていないおはじき。 |
| 手札 | 各プレイヤーが保持するタイル。上限 **6枚**。 |
| タイル袋（プール） | まだ誰の手札にも盤上にも無いタイルの集合。単一の共有プール。 |
| スコアボード | 各プレイヤーが獲得したおはじき（＝得点）を置く場所。得点＝おはじきの**枚数**（色は無関係）。 |
| 中立性 | 盤上のおはじき・タイルは所有者を持たない（誰の駒でもない）。 |
| 完全情報 | 両者の手持ち・手札・スコア・盤面はすべて相互に可視。 |

混色対応（AABイベントで使用）:

```
R + B → P（紫）
B + Y → G（緑）
R + Y → O（橙）
```

## 1. 構成要素と初期セットアップ

- ボード：5×5、初期は全セル空。
- おはじき：各プレイヤーに **赤4・青4・黄4＝12個**（両者で24個）。
- タイル：総24枚（紫8・緑8・橙8）。初期に各プレイヤーへ **袋からランダムに2枚ずつ**配る（=4枚が手札へ、残り20枚が袋）。
- スコアボード：各プレイヤー1枚（0点開始）。すべて公開。
- 先手／後手を決定（実装上は任意 or ランダム、UIで選択可）。

---

## 2. 手番の流れ（ターン）

プレイヤーは交互に手番を行う。1手番の手順は以下で固定：

1. **設置**：自分の手持ちから任意の1色のおはじきを1つ選び、**空きセル**1つに置く。（設置場所に制限なし）
2. **イベント解決**：直前に置いたおはじきによって新たに成立した並びを判定・解決（§3〜§6）。
3. 手番終了 → 相手の手番へ。

パスは不可。手持ちがある限り必ず1つ置く。

---

## 3. 並び（ライン）判定の基本規則

- 方向：**縦・横・斜め（4方向）**。斜めは `＼` と `／` の両方。
- ラインの単位は**連続する3セル**（ウィンドウ）。
- **判定対象は「直前に置いた駒を含むウィンドウ」のみ**（その駒によって引き起こされたものだけ）。
- 4個・5個連続した場合：**「置いた駒を含む、長さ3の連続ウィンドウ」を全て列挙し、重複なく各1回ずつ**イベント判定する。

  ウィンドウ列挙の定義（1方向あたり）：置いた駒を中心に、その方向で連続して埋まっているセル列を取り、長さ3の連続部分列のうち**置いた駒を含むもの全て**。
  - 例（横一列が全部埋まっている `c0 c1 c2 c3 c4` で `c2` に置いた）→ `(c0,c1,c2) (c1,c2,c3) (c2,c3,c4)` の **3ウィンドウ**。
  - 例（`c0 c1 c2 c3` の4連で端の `c3` に置いた）→ `(c1,c2,c3)` の **1ウィンドウ**。
  - 例（`c0 c1 c2 c3` の4連で `c1` に置いた）→ `(c0,c1,c2) (c1,c2,c3)` の **2ウィンドウ**。

- おはじきのウィンドウ（3つともおはじき）は §4/§5、タイルのウィンドウ（3つともタイル）は §6 で扱う。**おはじきとタイルが混在するウィンドウは何も起こさない。**

> 要最終承認①：上記「置いた駒を含む長さ3ウィンドウを全て各1回」で確定してよいか（ユーザー回答「初めて成立した3つ並びを被りなくカウント」の具体化）。

---

## 4. おはじき3並びイベント（おはじきのウィンドウ）

ウィンドウ内3つのおはじきの色構成は AAA / ABC / AAB の3通りのみ。

### 4-1. AAA（全同色）または ABC（全異色）→ タイル獲得
- 袋からランダムに**3枚**引いて手札に加える（ブラインド抽選、色は選べない）。
- 手札上限は6枚。超える分は引けない（例：手札4枚なら2枚だけ）。
- 袋の残数が3枚未満なら**引ける分だけ**（0枚なら何も起きない）。
- おはじきは盤上に残る（消えない）。

### 4-2. AAB（同色2＋異色1）→ 交換（得点化）
- ウィンドウ内に**2色**が存在する（多い色A×2、少ない色B×1）。
- アクティブプレイヤーが `mix(A,B)` のタイルを**手札に持っている場合のみ**、実行可能（**任意**：持っていても見送れる）。
- 実行時：
  - 少ない方（B×1）のおはじきを盤から取り除き、**アクティブプレイヤーのスコアボードへ**（+1点）。
  - そのセルに `mix(A,B)` タイルを置く（手札から消費）。
- 対応タイルを持っていない場合：交換不可。おはじきはそのまま盤に残り、手番は相手へ移る（得点なし）。

---

## 5. 複数ライン・タイル不足時の扱い

- 1手番で複数のおはじきウィンドウが同時成立し得る。**成立した各ウィンドウごとに個別にイベントを解決**（AAA/ABC はそれぞれタイル獲得、AABはそれぞれ交換＝それぞれ+1点）。
- **同一 `mix` タイルが必要な複数のAABウィンドウがあり、手札にそのタイルが1枚しかない**場合：どちらか好きな片方の交換にのみ使える（他方は成立させられない）。
- （要最終承認②）**「置いた駒自身が、複数方向で"少ない方(B)"になっている」**場合、取り除く対象セルが同一になるため物理的に一度しか交換できない。→ **プレイヤーがどれか1つのAABを選んで実行（+1点・タイル1枚設置）、残りは不成立**、とする。

---

## 6. タイル3並びイベント（タイルのウィンドウ）

- タイルが縦横斜めで3連続したら発生。**タイルの色は不問**（種類バラバラでもよい）。
- 判定対象は「この手番中に新たに盤へ置かれたタイル（§4-2の交換で設置されたもの）を含むウィンドウ」のみ。§3のウィンドウ列挙規則をタイルにも同様に適用（4/5連は重複なく複数成立）。
- 効果（各成立ウィンドウごとに1回）：
  - 相手のスコアが**1点以上**なら：相手スコアボードから**おはじき1個を自分のスコアボードへ移す**（相手 −1／自分 +1、色は不問）。
  - 相手のスコアが**0点**なら：相手の**手札タイルから好きな1枚**を選び自分の手札へ移す。ただし自分の手札上限6を超えるなら実行不可（何も起きない）。
  - 相手がスコア0かつ手札0なら：何も起きない。

---

## 7. 同一手番内のイベント解決順（正準順序）

1手番の「設置」後、以下の順で解決する（説明書FAQ「先に得点、後にタイル獲得」に準拠）：

```
1. 置いたおはじきを含む全おはじきウィンドウを、設置時点の盤面で列挙・分類（AAA/ABC/AAB）。
2. 【得点フェーズ】AABウィンドウの交換を解決（プレイヤーの選択・§5の競合規則を適用）。
   - 各交換：少ない方のおはじきを取り除き +1点、跡地に mix タイルを設置。
   - ※このフェーズで手に入る「まだ引いていないタイル」を交換に使うことはできない
     （交換に使えるのは設置前から手札にあったタイルのみ）。
3. 【タイル並びフェーズ】2で新たに設置されたタイルを含むタイルウィンドウを列挙し、§6を解決（得点/タイルの奪取）。
4. 【タイル獲得フェーズ】1で分類した AAA/ABC ウィンドウの袋抽選を解決（§4-1）。
5. 手番終了。
```

補足：
- 交換（§4-2）による得点化と、タイル並び（§6）による奪取は、どちらも「得点系」でありタイル袋抽選（§4-1）より前に解決する。
- ウィンドウの列挙は**設置直後の盤面**を基準に一括で行い、その後に解決する（解決中の駒除去・タイル設置が、既に列挙済みの他ウィンドウの「成立判定」を後から無効化しない）。ただし物理的競合（§5の同一セル）だけは解決時に調停する。

---

## 8. 終了条件・勝敗

- **終了**：両プレイヤーの手持ちおはじきが全て無くなったとき（＝24個すべて設置済み）。盤は1マス空く。
- **勝敗**：スコアボード上のおはじき枚数が多い方の勝ち。同数は**引き分け**。

## 9. データモデル（実装・AI共通の状態表現）

```
Color        = R | B | Y                    // おはじき
TileColor    = P | G | O                    // タイル（P=R+B, G=B+Y, O=R+Y）
CellContent  = Empty | Ohajiki(Color) | Tile(TileColor)

GameState {
  board:        CellContent[5][5]
  stock:        { [player]: { R:int, B:int, Y:int } }   // 手持ちおはじき残数
  hand:         { [player]: { P:int, G:int, O:int } }   // 手札タイル（合計 ≤ 6）
  bag:          { P:int, G:int, O:int }                  // タイル袋の残数（合計の初期値は 24 − 配布分）
  score:        { [player]: int }                        // スコアボード上のおはじき枚数
  toMove:       player                                   // 手番
  // 派生: gameOver = (stock[P0] 全0) && (stock[P1] 全0)
}
```

不変条件（テストで検証）:
- 各色おはじき総数保存：`盤上R + stock両者R + scoreに含まれるR... ` → **色別のおはじき保存は崩れる**（scoreは色を区別しないため）。代わりに**おはじき総数保存**：`盤上のおはじき数 + 両者stock合計 + 両者score合計 = 24`。
- タイル総数保存：`盤上のタイル数 + 両者hand合計 + bag合計 = 24`。
- `hand[player]合計 ≤ 6`、`score[player] ≥ 0`。

エンジンAPI（純関数・副作用なし）:

```
legalMoves(state) -> Move[]              // 「(色, 空きセル)」＋各AAB/§6選択の分岐
applyMove(state, move, choices) -> {states: [(prob, GameState)]}
                                         // 袋抽選のため確率分岐（チャンスノード）を返す
isTerminal(state) -> bool
result(state) -> {P0:int, P1:int}        // 終局時スコア
```

`Move` にはおはじきの設置（色＋座標）に加え、解決中のプレイヤー選択（どのAABを実行するか／§5競合の選択／§6でどのタイルを奪うか）を含める。設計上は「設置Move」→「解決サブ選択の連鎖」を1手番として扱う。

## 10. エッジケース一覧（実装チェックリスト）

1. 4連・5連での複数ウィンドウ成立（§3の列挙）。
2. 同一手番で複数AAB成立、同一 `mix` タイルが1枚のみ → 片方選択（§5）。
3. 置いた駒自身が複数方向で「少ない方」→ 1つのみ実行（要承認②）。
4. AAB交換で置いたタイルがタイル3並びを即成立 → 同手番で§6解決。
5. 交換で置いたタイルが複数のタイルウィンドウを同時成立 → それぞれ奪取。
6. §6で相手スコア0 → 相手手札から奪取。相手手札も0 → 不発。上限6超過 → 不発。
7. 袋残数 < 3 でAAA/ABC成立 → 引ける分だけ／0なら不発。
8. AAA/ABCが同手番で複数成立し、合計獲得が上限6を超える → 上限で打ち切り（引く順序は袋からランダムなので結果に差は出ないが、上限判定は逐次）。
9. 「得点化と袋抽選が同手番」→ 得点フェーズ→タイル並びフェーズ→袋抽選の順（§7）。引くタイルを交換に流用不可。
10. 終局判定：24個設置直後。最後の設置で発生したイベントも通常どおり解決してから終局。

## 11. AI設計方針（フェーズ③、参考）

- 本ゲームは**完全情報＋既知分布の確率要素（袋抽選）**のみ。隠し情報は無い → ISMCTSは不要。
- 適合手法：**Expectimax / チャンスノード付きMCTS（あるいは両者の併用）**。
  - MAX/MINノード＝各プレイヤーの着手選択、CHANCEノード＝袋抽選（分岐確率は袋残数から厳密に算出）。
- 高速化：ビットボード表現、着手・並び判定のインクリメンタル更新、置換表、袋状態のハッシュ化。
- 強化路線（将来）：自己対戦データ＋価値/方策ネットワーク（AlphaZero系）も選択肢。完全情報なので学習・評価が安定。
- そのため**エンジンは高速言語（C++）**で実装し、大量シミュレーションに耐える構成にする（§12）。

## 12. 技術構成・フェーズ計画（参考／別途合意）

- コア（ルールエンジン＋AI）：**C++** → Web用に **Emscripten/WASM**、学習・自己対戦用に**ネイティブ**ビルド。
  - ビルド：**CMake** で 2 ターゲット（`emcmake` による WASM ／ ネイティブ）。
  - JS連携：Emscripten **embind**、または薄い **C ABI + JSラッパ**で `GameState`/`Move` を受け渡し。
- 描画：**TypeScript + Three.js / react-three-fiber**（WebGL）。タイルのマーブル表現・並び成立・交換・奪取のアニメーション。
- フェーズ：
  1. **プレイ可能なGUI**（ルール完全実装＋美麗レンダリング＋アニメ）
  2. **簡単なBot**（貪欲＋浅いExpectimax）
